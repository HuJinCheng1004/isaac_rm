## §1 — Register task and set up the scene (robot, objects)

### Description

Tasks register via `gymnasium.register` in a per-task `__init__.py`, pointing `env_cfg_entry_point` at a `ManagerBasedRLEnvCfg` subclass. The scene is declared as a `@configclass InteractiveSceneCfg` subclass containing the robot (`ArticulationCfg`), rigid objects (`RigidObjectCfg`), static props (`AssetBaseCfg`), ground plane, and dome light. The robot asset is loaded from a local URDF via `UrdfFileCfg`; rigid props use USD files from IsaacLab Nucleus. All per-env assets use `{ENV_REGEX_NS}/<name>` prim paths. The `ManagerBasedRLEnvCfg.__post_init__` sets simulation timing (`decimation`, `episode_length_s`, `sim.dt`) and PhysX solver parameters.

### File pointers

| What | Where |
|------|-------|
| Registration entry | `source/chassis_nav/chassis_nav/tasks/push/__init__.py` |
| Scene config class | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:RMPushSceneCfg` |
| Robot config | `source/chassis_nav/chassis_nav/robots/arm.py:ARM_PUSH_CFG` |
| Robot joint constants | `source/chassis_nav/chassis_nav/robots/arm.py` (`EE_BODY_NAME`, `ARM_JOINT_NAMES`) |
| Object prop (DexCube) | `ISAAC_NUCLEUS_DIR/Props/Blocks/DexCube/dex_cube_instanceable.usd` (Nucleus) |
| Table prop | `ISAAC_NUCLEUS_DIR/Props/Mounts/SeattleLabTable/table_instanceable.usd` (Nucleus) |
| Reference example end-to-end | `source/chassis_nav/chassis_nav/tasks/push/` |

### Code template

```python

    id="Isaac-<TaskName>-v0",
    entry_point="isaaclab.envs:ManagerBasedRLEnv",

        "skrl_cfg_entry_point": f"{agents.__name__}:skrl_ppo_cfg.yaml",
    },
)

# <task>/<task>_env_cfg.py  — scene

@configclass

    object: RigidObjectCfg = RigidObjectCfg(
        prim_path="{ENV_REGEX_NS}/Object",
        init_state=RigidObjectCfg.InitialStateCfg(pos=<OBJECT_INIT_POS>, rot=[1,0,0,0]),
        spawn=UsdFileCfg(usd_path=f"{ISAAC_NUCLEUS_DIR}/Props/Blocks/DexCube/dex_cube_instanceable.usd",
                         scale=(0.8,0.8,0.8), rigid_props=RigidBodyPropertiesCfg(...)),
    )
    table = AssetBaseCfg(prim_path="{ENV_REGEX_NS}/Table", ...)
    plane = AssetBaseCfg(prim_path="/World/GroundPlane", ...)
    light = AssetBaseCfg(prim_path="/World/light", spawn=sim_utils.DomeLightCfg(...))

@configclass
class <TaskName>EnvCfg(ManagerBasedRLEnvCfg):
    scene: <TaskName>SceneCfg = <TaskName>SceneCfg(num_envs=1024, env_spacing=2.5)
    # ... other manager fields ...
    def __post_init__(self):
        self.decimation = 2
        self.episode_length_s = 200.0 / 30.0
        self.sim.dt = 1.0 / 60.0
        self.sim.render_interval = self.decimation
        self.sim.physx.bounce_threshold_velocity = 0.01
```

- Object placement: initial position and spawn jitter range (table center for push tasks: `pos=[-0.5, -0.2, 0.055]`)
- Table position: must align with robot's reachable workspace (`[-0.5, -0.2, 0.0]` for -x-side arm)
- `num_envs` for training (canonical: 1024; PPO parallel config default: 2048)

## §2 — Action types (delta / absolute joint position, delta / absolute end-effector pose)

### Description

This benchmark uses `mdp.JointPositionActionCfg` for **absolute joint-position** control. The `ActionsCfg` dataclass holds one or more `ActionTerm` fields; the robot's active joints are filtered by name via `joint_names`. For the canonical push task, only the 6 right-arm joints (`r_joint1`–`r_joint6`) are controlled; all other joints (chassis, lift, head, left arm, gripper) are locked by high-stiffness implicit actuators in `ARM_PUSH_CFG`. Delta EE-pose (`DifferentialInverseKinematicsActionCfg`) is available in IsaacLab but not wired in this benchmark's canonical task.

### File pointers

| Action mode | Class / function | Path |
|-------------|------------------|------|
| Absolute joint position | `mdp.JointPositionActionCfg` | `isaaclab.envs.mdp.actions` (library) |
| Delta joint position | `mdp.RelativeJointPositionActionCfg` | `isaaclab.envs.mdp.actions` (library; not used in canonical) |
| Absolute EE pose | n/a | not wired in this benchmark |
| Delta EE pose | `mdp.DifferentialInverseKinematicsActionCfg` | library; requires IK solver config — not wired |
| Gripper / binary | n/a | gripper locked closed in `ARM_PUSH_CFG` (high-stiffness actuator) |

### Code template

```python
# ActionsCfg — absolute joint-position on the 6 active arm joints
_ARM_CFG = SceneEntityCfg("robot", joint_names=ARM_JOINT_NAMES)  # ["r_joint1".."r_joint6"]

@configclass
class ActionsCfg:
    arm_action = mdp.JointPositionActionCfg(
        asset_name="robot",
        joint_names=ARM_JOINT_NAMES,   # 6 DOF right arm only
        scale=0.5,                     # output multiplied before adding to default offset
        use_default_offset=True,       # actions are relative to init_state joint_pos
    )
    # No gripper action — gripper stays closed (palm-face pusher)
```

### Decisions the future agent must resolve

- Which joints to control: confirm `ARM_JOINT_NAMES = ["r_joint1",.."r_joint6"]` is correct for the new task, or override
- `scale`: 0.5 is the canonical default; smaller values give finer but slower arm movement

## §3 — Initialize / reset the task

### Description

Reset logic lives in `EventCfg` entries with `mode="reset"`. Each entry names a library function from `isaaclab.envs.mdp` and a parameter dict. Two reset terms are canonical: one randomizes robot joint positions around the default home pose (`reset_joints_by_offset`), and one randomizes the object position around the table center (`reset_root_state_uniform`). The `asset_cfg` parameter narrows which joints or bodies are touched; `_ARM_CFG = SceneEntityCfg("robot", joint_names=ARM_JOINT_NAMES)` restricts the joint reset to the 6 active arm joints so locked joints are untouched.

### File pointers

| What | Where |
|------|-------|
| Reset hook | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:EventCfg` (mode="reset" terms) |
| Joint reset function | `isaaclab.envs.mdp:reset_joints_by_offset` (library) |
| Object reset function | `isaaclab.envs.mdp:reset_root_state_uniform` (library) |

### Code template

```python
@configclass
class EventCfg:
    # §3 — Reset arm joints with uniform offset from home pose
    reset_robot_joints = EventTerm(
        func=mdp.reset_joints_by_offset,
        mode="reset",
        params={
            "position_range": (-0.3, 0.3),   # rad — tune per task
            "velocity_range": (0.0, 0.0),
            "asset_cfg": SceneEntityCfg("robot", joint_names=ARM_JOINT_NAMES),
        },
    )
    # §3 — Randomize object position on table surface
    reset_object_position = EventTerm(
        func=mdp.reset_root_state_uniform,
        mode="reset",
        params={
            "pose_range": {"x": (-0.05, 0.05), "y": (-0.05, 0.05), "z": (0.0, 0.0)},
            "velocity_range": {},
            "asset_cfg": SceneEntityCfg("object"),
        },
    )
    # §7 DR terms also live in EventCfg with mode="startup" — see §7
```

### Decisions the future agent must resolve

- Joint reset range: `(-0.3, 0.3)` rad is canonical; widen for harder exploration, narrow for gentler starts
- Object jitter range: `±0.05 m` around table center; confirm the center `[-0.5, -0.2]` is reachable

## §4 — Goal and termination

### Description

The goal is a 2D target position on the table surface, expressed as a `UniformPoseCommandCfg` named `object_pose`. It resamples every 4 s (so the agent sees multiple goal positions per episode) and is included directly in the observation. The target z is pinned to the block resting height (`0.055 m`) making it a planar-push goal. Termination has two conditions: `time_out` (200 control steps = ~6.67 s) and `object_dropping` (block falls below table level). Success is detected by the `block_at_goal` reward term which also sets `env._reach_success` for harbor's `eval/success_rate` metric.

### File pointers

| What | Where |
|------|-------|
| Goal sampler | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:CommandsCfg` |
| Termination terms | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:TerminationsCfg` |
| Success metric | `source/chassis_nav/chassis_nav/tasks/push/mdp/rewards.py:block_at_goal` (sets `env._reach_success`) |

### Code template

```python
@configclass
class CommandsCfg:
    object_pose = mdp.UniformPoseCommandCfg(
        asset_name="robot",
        body_name=EE_BODY_NAME,             # "r_link8" — debug-vis attachment only
        resampling_time_range=(4.0, 4.0),
        debug_vis=True,
        ranges=mdp.UniformPoseCommandCfg.Ranges(
            pos_x=(-0.75, -0.25),           # arm's reachable x range (robot-frame)
            pos_y=(-0.40, 0.00),            # arm's reachable y range
            pos_z=(0.055, 0.055),           # fixed at block resting height
            roll=(0.0, 0.0), pitch=(0.0, 0.0), yaw=(0.0, 0.0),
        ),
    )

@configclass
class TerminationsCfg:
    time_out = DoneTerm(func=mdp.time_out, time_out=True)
    object_dropping = DoneTerm(
        func=mdp.root_height_below_minimum,
        params={"minimum_height": -0.05, "asset_cfg": SceneEntityCfg("object")},
    )
```

### Decisions the future agent must resolve

- Goal xy range: must lie within the arm's reachable workspace (`x∈[-0.75,-0.25]`, `y∈[-0.40,0.00]` in robot root frame)
- `resampling_time_range`: `(4.0, 4.0)` fixed cadence; widen for curriculum
- Success threshold: `0.05 m` (5 cm) is canonical; adjust per task difficulty

### Smoke check


## §5 — Observation

### Description

A single `PolicyCfg` observation group with `concatenate_terms=True` and `enable_corruption=True` (additive uniform noise on joint pos/vel). Terms in concatenation order: right-arm joint positions relative to default (6), right-arm joint velocities relative to default (6), block position in robot root frame (3, task-local function), goal command pos+quat (7, `UniformPoseCommandCfg` output), last action (6). Total = **28**. The task-local observation function `object_position_in_robot_root_frame` lives in `mdp/observations.py`.

### File pointers

| Obs term | Source |
|----------|--------|
| Joint pos / vel | `isaaclab.envs.mdp:joint_pos_rel`, `joint_vel_rel` (library) |
| Object position in robot frame | `source/chassis_nav/chassis_nav/tasks/push/mdp/observations.py:object_position_in_robot_root_frame` |
| Goal command | `isaaclab.envs.mdp:generated_commands` with `command_name="object_pose"` (library) |
| Last action | `isaaclab.envs.mdp:last_action` (library) |
| ObservationsCfg | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:ObservationsCfg` |

### Code template

```python
@configclass
class ObservationsCfg:
    @configclass
    class PolicyCfg(ObsGroup):
        joint_pos = ObsTerm(
            func=mdp.joint_pos_rel,
            params={"asset_cfg": SceneEntityCfg("robot", joint_names=ARM_JOINT_NAMES)},
            noise=Unoise(n_min=-0.01, n_max=0.01),
        )
        joint_vel = ObsTerm(
            func=mdp.joint_vel_rel,
            params={"asset_cfg": SceneEntityCfg("robot", joint_names=ARM_JOINT_NAMES)},
            noise=Unoise(n_min=-0.01, n_max=0.01),
        )
        object_position = ObsTerm(func=mdp.object_position_in_robot_root_frame)
        target_object_position = ObsTerm(
            func=mdp.generated_commands, params={"command_name": "object_pose"}
        )
        actions = ObsTerm(func=mdp.last_action)

        def __post_init__(self):
            self.enable_corruption = True
            self.concatenate_terms = True

    policy: PolicyCfg = PolicyCfg()
```

Task-local observation function in `mdp/observations.py`:
```python
def object_position_in_robot_root_frame(env, robot_cfg=SceneEntityCfg("robot"),
                                         object_cfg=SceneEntityCfg("object")):
    """Block position expressed in the robot root frame (3D)."""
    object_pos_w = env.scene[object_cfg.name].data.root_pos_w[:, :3]
    object_pos_b, _ = subtract_frame_transforms(
        env.scene[robot_cfg.name].data.root_pos_w,
        env.scene[robot_cfg.name].data.root_quat_w, object_pos_w)
    return object_pos_b
```

- Obs total dim: 6+6+3+7+6 = **28**; if the new task changes action dim or adds obs terms, update accordingly
- Noise levels: `Unoise(-0.01, 0.01)` on joint pos/vel; increase for harder sim-to-real

## §6 — Reward (read by `reward-generator`)

### Description

Reward is composed by IsaacLab's `RewardManager` as a **weighted sum** of term functions: `total = Σ weight_i × dt × func_i(env)`. Each term is a `RewTerm(func=..., params=..., weight=...)` entry in `RewardsCfg`. The `dt` multiplication is implicit in the manager (no manual `weight /= step_dt` needed). Task-local reward functions live in `mdp/rewards.py`; regularizer terms (`action_rate_l2`, `joint_vel_l2`) come from the IsaacLab library. Per-term episodic returns are logged to `metrics.jsonl` by the harbor train script via `reward_manager._episode_sums` — no separate `/harbor:reward-add-log` wiring needed for this benchmark.

### File pointers

| What | Where |
|------|-------|
| Reward composition root | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:RewardsCfg` |
| Task-local reward functions | `source/chassis_nav/chassis_nav/tasks/push/mdp/rewards.py` |
| Composition operator | **sum** (IsaacLab `RewardManager`) |
| Reference task with rich reward | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py` (see `RewardsCfg` for full term list) |

### Code template

```python
@configclass
class RewardsCfg:
    # Positive shaping — pull EE toward the block (tanh kernel)
    reaching_block = RewTerm(
        func=mdp.object_ee_distance_body,
        params={"std": <STD>, "robot_cfg": SceneEntityCfg("robot", body_names=[EE_BODY_NAME])},
        weight=<WEIGHT>,
    )
    # Positive shaping — pull block toward goal (tanh kernel, coarse)
    block_to_goal_tracking = RewTerm(
        func=mdp.block_to_goal_distance,
        params={"std": 0.3, "command_name": "object_pose"},
        weight=<WEIGHT>,
    )
    # Success indicator — sets env._reach_success for harbor eval tracking
    success = RewTerm(
        func=mdp.block_at_goal,
        params={"threshold": 0.05, "command_name": "object_pose"},
        weight=1.0,   # must be > 0; harbor derives success_rate = mean(success) / weight
    )
    # Regularizers
    action_rate = RewTerm(func=mdp.action_rate_l2, weight=-1e-4)
    joint_vel   = RewTerm(func=mdp.joint_vel_l2, weight=-1e-4,
                          params={"asset_cfg": SceneEntityCfg("robot", joint_names=ARM_JOINT_NAMES)})
```

Available task-local reward functions (see `mdp/rewards.py` for full signatures):
- `object_ee_distance_body(env, std, robot_cfg)` — tanh reaching reward, EE → block CoM
- `ee_z_to_block(env, std, robot_cfg)` — exp height-alignment reward, pulls EE z to block z
- `block_to_goal_distance(env, std, command_name)` — tanh tracking reward, block → goal
- `block_at_goal(env, threshold, command_name)` — binary success; also sets `env._reach_success`

### Decisions the future agent must resolve

- Primary reaching term: `reaching_block` std and weight — larger std gives gradient at farther distances
- Whether to add `ee_z_to_block` for height alignment (needed when arm approaches from above)
- Tracking weight vs reaching weight balance — high tracking weight creates "free reward" exploit

## §7 — Domain randomization (read by `dr-generator`)

### Description

DR in this benchmark uses `EventCfg` entries with `mode="startup"` — each term runs once per env instance at simulation startup and stays fixed for the env lifetime (i.e. per-env randomization, not per-episode). The two canonical DR axes are: (1) friction on robot finger contact surfaces via `randomize_rigid_body_material`, and (2) block mass via `randomize_rigid_body_mass`. Both functions are IsaacLab library terms. There is no `mode="interval"` DR in the canonical task. The DR terms coexist in `EventCfg` alongside the `mode="reset"` terms from §3.

### File pointers

| DR axis | Term / function | Path |
|---------|-----------------|------|
| Finger friction | `mdp.randomize_rigid_body_material` | `isaaclab.envs.mdp` (library) |
| Block mass | `mdp.randomize_rigid_body_mass` | `isaaclab.envs.mdp` (library) |
| Joint stiffness / damping | n/a | not wired in canonical task |
| Observation noise | via `Unoise` in `ObservationsCfg` (§5) | `push_env_cfg.py:ObservationsCfg.PolicyCfg` |
| Action noise | n/a | not wired |
| Sensor / lighting | n/a | no visual pipeline |
| Reference task with DR | `source/chassis_nav/chassis_nav/tasks/push/push_env_cfg.py:EventCfg` (startup terms) |

### Code template

```python
# Add to EventCfg alongside the §3 reset terms:

physics_material = EventTerm(
    func=mdp.randomize_rigid_body_material,
    mode="startup",
    params={
        "asset_cfg": SceneEntityCfg("robot", body_names=["r_Link_finger1", "r_Link_finger2"]),
        "static_friction_range": (0.8, 1.2),
        "dynamic_friction_range": (0.8, 1.2),
        "restitution_range": (0.0, 0.0),
        "num_buckets": 64,
    },
)
# §7 DR — block mass ±20% multiplicative (startup, per-env-instance)
block_mass = EventTerm(
    func=mdp.randomize_rigid_body_mass,
    mode="startup",
    params={
        "asset_cfg": SceneEntityCfg("object"),
        "mass_distribution_params": (0.8, 1.2),
        "operation": "scale",
    },
)
```

### Decisions the future agent must resolve

- Which contact bodies to randomize friction on: `["r_Link_finger1", "r_Link_finger2"]` for push tasks; adjust if using a different EE
- Mass range: `(0.8, 1.2)` scale is canonical ±20%; widen for harder sim-to-real
- Whether to add `mode="reset"` DR (per-episode) for motor gains or additional friction — not in canonical task

## Authoring conventions for downstream agents

- **Mirror, don't invent.** The canonical example file is the source of truth. If a pattern is not in this doc but you find it in the example file, prefer the example.
- **Resolve ambiguity by asking the user once.** The "Decisions" sub-blocks list the questions that need user input. Batch them into a single `AskUserQuestion` per section, not one question at a time.
- **Per-section smoke is fatal.** A section is not "done" until its smoke check passes. Do not move to the next section on a smoke failure — diagnose, patch, retry, or escalate.
- **Stay surgical.** Adding a new task should add files under one new directory and one registration line. Do not refactor sibling tasks.
- **Workspace awareness.** The RM robot's right arm extends toward -x when at home. Place objects and goals in the -x side (`x∈[-0.75,-0.25]`, `y∈[-0.40,0.00]`) of the robot root frame. The lift platform is locked at `platform_joint=0.5 m` — the arm's vertical reach is fixed.
- **EE body name.** Always use `EE_BODY_NAME = "r_link8"` (imported from `chassis_nav.robots.arm`). Do not hardcode the string.
